## 2. 产品概述

### 2.1 项目定位

SmartMall 是一个面向毕业设计场景的数码电子商城系统，包含移动端用户商城、后台管理端、后端接口服务与数据库支撑。系统核心目标不是上线生产，而是完整展示典型电商业务链路，并体现活动营销、会员体系、订单流转、后台运营配置等综合能力。

### 2.2 产品目标

1. 支撑完整演示链路  
   覆盖用户登录、浏览商品、加购、结算、支付、下单、订单管理、活动参与、后台配置等完整流程。

2. 支撑论文与答辩表达  
   模块边界清晰，功能逻辑完整，能对应需求分析、系统设计、数据库设计、功能实现与测试章节。

3. 支撑后续按优先级持续迭代  
   用正式 PRD 统一当前事实、待做项、优先级和验收标准，避免继续零散追加需求。

### 2.3 目标用户

- 普通用户：浏览、购买、参与拼团/秒杀、管理订单与资产
- VIP 用户：享受会员折扣、余额/积分权益与更强的账户展示
- 管理员：管理商品、订单、活动、评论、优惠券、会员与看板数据

### 2.4 非目标

- 不接入真实支付通道
- 不实现生产级高并发架构与风控能力
- 不实现跨端商业化部署方案
- 不做多租户、权限细粒度隔离、企业级审计系统

## 3. 当前系统现状

### 3.1 总体架构现状

- 前端：
  - 移动端：Vue 3 + Vant
  - 管理端：Vue 3 + Element Plus
- 后端：Flask + SQLAlchemy + PyMySQL
- 数据库：MySQL
- 数据交互方式：移动端与管理端均通过后端统一 REST 接口获取和提交数据
- 图片资源：后端 `uploads` 目录托管本地图片资源

### 3.2 用户端已实现能力

1. 账号体系
   - 注册、登录、切换账号
   - 用户头像与昵称修改
   - 会员文案单独编辑并与当前用户绑定

2. 首页与商品浏览
   - 首页轮播图
   - 商品列表展示
   - 分类筛选与搜索
   - 快捷入口拖拽排序
   - 商品详情查看

3. 购物行为
   - 收藏/取消收藏
   - 加入购物车
   - 购物车数量调整、删除、勾选结算
   - 单商品直接购买

4. 结算与支付
   - 地址管理与选择
   - 优惠券领取、查看、结算使用
   - 积分抵扣
   - 余额充值
   - 扫码支付流程演示
   - 支付成功页

5. 订单与售后
   - 我的订单
   - 查看全部订单
   - 订单详情
   - 物流轨迹展示
   - 取消订单
   - 确认收货
   - 订单评价
   - 联系客服 / 拨打电话

6. 活动与会员
   - VIP 会员升级与会员折扣
   - 拼团大厅、开团、参团、拼团码
   - 秒杀大厅、秒杀倒计时、秒杀详情状态展示

### 3.3 管理端已实现能力

1. 登录与框架
   - 管理员登录
   - 左侧菜单 + 页面容器

2. 运营管理
   - 数据看板
   - 轮播图管理
   - 商品管理
   - 订单管理
   - 评论管理
   - 优惠券管理
   - 会员管理

3. 活动管理
   - 拼团规则设置
   - 拼团商品设置页
   - 秒杀规则设置
   - 秒杀商品设置页
   - 商品管理页中对拼团/秒杀信息进行展示和配置联动

### 3.4 后端现状

- 后端提供统一接口给移动端和管理端使用
- 当前已支持商品、订单、评论、会员、拼团、秒杀、优惠券、收藏、地址、签到、充值等业务
- 秒杀商品已具备专用字段：
  - `is_seckill`
  - `seckill_price`
  - `seckill_stock`
  - `seckill_limit_per_user`
  - `seckill_start_at`
  - `seckill_end_at`
- 秒杀订单已具备标记字段：
  - `is_seckill_order`
- 拼团当前仍以全局配置为主：
  - 成团人数
  - 折扣比例
- 后端测试已覆盖主要接口和秒杀场景

### 3.5 当前已知问题

1. 拼团系统仍以全局规则驱动  
   当前没有商品级拼团参数，拼团价格与成团人数依赖全局配置，灵活性不如秒杀系统。

2. 拼团超时与失败状态仍需进一步收严  
   目前拼团流程可演示，但超时检测与状态更新机制仍有继续增强空间。

3. 购物车结算优惠券规则仍偏简化  
   当前更接近总价门槛校验，未细化到品类/单品适用规则。

4. 项目对本地数据库环境有依赖  
   当前默认连接本机 MySQL，跨电脑迁移时需要单独处理数据库环境与数据迁移。

5. 仓库当前存在脏工作区  
   后续执行必须按分支和 commit 粒度整理，不能在脏状态下混做功能。

## 4. 需求范围与阶段目标

PRD 执行分为 P0、P1、P2 三阶段，按优先级推进，不跨阶段混做。

### 4.1 P0 核心稳定版

目标：修复影响演示和主流程正确性的关键问题，确保整套系统“能讲、能演、能跑通”。

优先覆盖：

- 秒杀规则闭环与限购正确性
- 拼团主链路稳定性
- 订单、支付、结算主流程可演示
- 后台活动配置与商品管理一致性
- 页面关键交互明显 Bug 修复

验收结果：

- 主要链路能完整演示
- 后台配置与前台展示可对应
- 构建通过、后端测试通过、关键流程可手测

### 4.2 P1 完整度增强版

目标：补足毕业设计系统在“功能完整性”上的缺口，让产品更像一套完整商城系统。

优先覆盖：

- 默认地址
- 修改密码
- 新人券
- 拼团大厅实时状态
- 更完整的活动配置与展示闭环

验收结果：

- 用户端账户与地址能力更完整
- 活动系统不只可演示，还具备更强的运营闭环表达

### 4.3 P2 展示与加分版

目标：提升答辩观感、作品完成度与展示说服力。

优先覆盖：

- 搜索筛选增强
- 商品分享能力
- 消息提示
- 看板趋势与排行补充
- 前后端体验统一打磨

验收结果：

- 页面观感更成熟
- 论文/答辩中可展示更多“扩展能力”和“设计深度”

## 6. 重点活动系统专项

### 6.1 拼团专项

#### 当前定位

拼团目前采用 **全局拼团规则**，即：

- 成团人数来自系统配置
- 拼团折扣来自系统配置
- 商品是否参与拼团由分类中的“拼团”标记决定

当前默认不引入商品级拼团字段，先保证全局拼团方案稳定可演示。  
后续是否演进到商品级拼团配置，放入 P1 评估，不在本版 PRD 中强制执行。

#### 业务规则

- 用户可在拼团大厅查看拼团商品
- 用户可选择：
  - 开团
  - 参团
  - 直接购买
  - 退出拼团流程
- 开团后生成拼团码
- 参团需输入有效拼团码
- 达到系统配置人数即成团成功
- 拼团价 = 商品原价 × 全局拼团折扣

#### 状态流转

- 开团后未成团：订单状态为拼团中
- 成团成功：进入正常后续订单状态
- 超时未成团：进入拼团失败/关闭状态，并按规则处理订单

#### 关键风险

- 拼团超时检测当前仍有增强空间
- 实时大厅展示能力仍需补足

#### 拼团验收

- 用户能完成开团、参团、支付
- 拼团码可展示、可输入、可生效
- 成团规则与后台配置一致
- 拼团中的订单在用户端可见

### 6.2 秒杀专项

#### 商品识别规则

秒杀商品满足任一条件即进入秒杀逻辑：

- 商品显式配置 `is_seckill = true`
- 或商品分类中含“秒杀”

#### 核心参数

- 秒杀价
- 秒杀库存
- 每人限购数量
- 开始时间
- 结束时间

#### 前端展示规则

- 秒杀大厅按状态展示商品
- 商品详情展示：
  - 秒杀价
  - 原价
  - 状态标签
  - 剩余库存
  - 限购数量
  - 活动时间

#### 后端下单规则

- 非活动时间禁止秒杀下单
- 库存不足禁止下单
- 超出限购禁止下单
- 秒杀商品不可走购物车加购
- 秒杀商品不可使用优惠券和积分
- 秒杀订单记录秒杀标记

#### 超时与库存处理

- 秒杀库存扣减和状态更新由后端处理
- 超时关闭和库存恢复在现有实现上继续验证和增强

#### 管理端入口

- 秒杀规则设置页：维护全局支付时限
- 秒杀商品设置页：维护商品级秒杀参数
- 商品管理页：同步展示秒杀信息

#### 秒杀验收

- 后台配置后前台可读到秒杀状态
- 同一用户不能突破限购
- 秒杀商品不能加入购物车
- 秒杀流程可完整演示并生成订单

## 7. 非功能要求

### 7.1 兼容环境

- 开发环境：Windows
- IDE：VS Code / PyCharm
- 统一要求：当前两种 IDE 都只作为代码编辑和启动工具，不改变业务数据来源

### 7.2 启动方式

- 当前依赖本机 MySQL 数据库
- 默认数据库连接来自后端配置
- 跨电脑运行时需单独准备数据库环境或迁移数据

- 主流程必须可录屏演示
- 主流程必须可现场答辩操作

- 旧数据兼容风险
- 脏工作区造成版本污染风险
- 活动逻辑复杂度增加后的联动风险
- IDE 切换导致环境不一致风险

## 9. git 执行与版本管理规则

### 9.1 PRD 阶段

- 从当前基线切出：`feature/prd-v1`
- 该分支只提交 PRD 文档
- commit 规范：
  - `docs: 新增 SmartMall PRD v1`
  - `docs: 完善 P0/P1/P2 需求与验收标准`

### 9.2 开发阶段

PRD 确认后按阶段开分支：

- `feature/p0-stability`
- `feature/p1-completeness`
- `feature/p2-polish`

每个分支只做对应阶段事项。

### 9.3 commit 规范

按功能点拆分提交，例如：

- `feat: 完善秒杀后端校验与限购`
- `fix: 修复拼团大厅交互与状态展示`
- `style: 优化移动端首页首屏视觉`
- `docs: 更新活动系统说明与验收标准`

### 9.4 合并顺序

1. `feature/prd-v1`
2. `feature/p0-stability`
3. `feature/p1-completeness`
4. `feature/p2-polish`

### 9.5 当前仓库基线要求

- 当前仓库为脏工作区
- 不允许在未识别基线的情况下直接混做阶段开发
- 文档分支优先落地 PRD，作为后续整理与执行依据
